## Bug

Colored H1 headings (per-chapter color theme) are only rendered with color in the PDF export. In the frontend — both the page editor preview and the Preview tab — H1 headings appear without the chapter color background. The color is set on the chapter but not propagated to the markdown renderer when displaying text slots.

## Expected behavior

When a chapter has a color set, all `# Heading` elements in text slots of that chapter's sections should render with a colored background and white text — matching the PDF output (colorbox with background color, white bold text).

## Where to fix

### 1. Markdown renderer
`web/src/utils/markdown.ts` — the `MarkdownContent` component or the markdown-to-HTML conversion needs to accept an optional `chapterColor` parameter. When provided, `<h1>` elements should be styled with:
- `background-color: {chapterColor}`
- `color: white`
- `font-weight: bold`
- `padding: 10px 12px` (matching the PDF padding from the related task)
- `font-family: Source Sans 3`

### 2. Page slot rendering
`web/src/pages/BookEditor/PageSlot.tsx` — when rendering a text slot, the chapter color needs to be passed down to the markdown renderer. The slot needs to know which section → which chapter → what color.

### 3. Preview tab
`web/src/pages/BookEditor/PreviewTab.tsx` and `PreviewModal.tsx` — same as above, chapter color must be propagated to text slot rendering.

### 4. WYSIWYG preview
If the text editing dialog/preview shows rendered markdown, it also needs the chapter color.

## How to get chapter color for a slot

The chain is: page → page.section_id → section.chapter_id → chapter.color. The book data (from `useBookData`) contains all chapters with their colors. Build a lookup: section_id → chapter_color and pass it through to wherever text slots are rendered.
