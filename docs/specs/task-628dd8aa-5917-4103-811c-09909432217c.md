# Persist Text Check Results in Database

Text check results (from AI "Kontrola textu") are currently stored only in React component state and lost on page refresh. They must be persisted so users see which texts have been checked, which are clean, and which have errors.

## Requirements

### 1. New database table

Create a `text_check_results` table:

```sql
CREATE TABLE text_check_results (
  id SERIAL PRIMARY KEY,
  source_type TEXT NOT NULL,          -- 'section_photo' or 'page_slot'
  source_id TEXT NOT NULL,            -- section_id:photo_uid or page_id:slot_index
  field TEXT NOT NULL,                -- 'description', 'note', or 'text_content'
  content_hash TEXT NOT NULL,         -- SHA-256 of the text that was checked
  status TEXT NOT NULL,               -- 'clean' or 'has_errors'
  readability_score INTEGER,          -- 0-100
  corrected_text TEXT,                -- corrected version (if errors found)
  changes JSONB,                      -- array of change descriptions
  cost_czk NUMERIC(10,4),            -- cost of the check
  checked_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE UNIQUE INDEX idx_text_check_source ON text_check_results(source_type, source_id, field);
```

The `content_hash` field tracks which version of the text was checked. If the text changes after checking, the check is stale and should show as "unchecked" again.

### 2. Backend: store results after check

When `POST /api/v1/text/check` is called, the caller should also be able to persist the result. Two approaches:

**Option A (preferred):** Add a new endpoint `POST /api/v1/text/check-and-save` that accepts `source_type`, `source_id`, `field`, and `text` — runs the check AND saves the result to the database. This way the frontend makes one call instead of two.

**Option B:** Add a separate `POST /api/v1/text/check-results` endpoint to save results after checking.

### 3. Backend: return check status with texts

Add check status to the book detail or a dedicated endpoint:
- `GET /api/v1/books/{id}/text-check-status` — returns a map of `source_type:source_id:field` → `{ status, readability_score, checked_at, is_stale }` for all texts in the book
- `is_stale` is `true` when the current text content hash doesn't match `content_hash` in the stored result (text was edited after checking)

### 4. Frontend: load and display persisted status

On the Texts tab:
- On mount, fetch check status for the book
- Display persisted status on each text row (clean / has errors / stale / unchecked)
- Stale results (text changed after check) should show as "Změněno od kontroly" with a different color (amber)
- Stats cards should reflect persisted state, not just session state
- When running "Zkontrolovat vše", update the DB after each check completes
- When running individual text check from PhotoDescriptionDialog or TextSlotDialog, also persist the result

### 5. Batch check integration

The "Zkontrolovat vše" button should:
- Skip texts that are already checked and not stale (saves money)
- Only re-check stale texts and unchecked texts
- Update persisted status as each check completes

### 6. i18n

Add keys:
- `books.editor.staleCheck`: "Změněno od kontroly" (cs) / "Changed since check" (en)
