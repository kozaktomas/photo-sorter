## Problem

The MCP server runs as a separate `mcp-serve` command on its own port (8086). This means running two processes, two ports, duplicate initialization (PostgreSQL, PhotoPrism client). It should be part of the existing `serve` command.

## Solution

Mount the MCP SSE server as a sub-route on the existing Chi router under `/mcp/`. The MCP server starts automatically when `MCP_API_TOKEN` is set — if the env var is missing, MCP routes are simply not registered (no error, just skipped).

## Implementation

### Step 1: Extend web.Server to optionally include MCP

In `internal/web/server.go`, add an optional MCP server field:

```go
type Server struct {
    config         *config.Config
    router         *chi.Mux
    httpServer     *http.Server
    jobManager     *handlers.JobManager
    sessionManager *middleware.SessionManager
    mcpServer      *mcp.Server  // nil if MCP_API_TOKEN not set
}
```

Add a setter or constructor option to pass the MCP server instance.

### Step 2: Mount MCP routes in routes.go

In `internal/web/routes.go`, if `s.mcpServer` is not nil, mount it:

```go
if s.mcpServer != nil {
    s.router.Route("/mcp", func(r chi.Router) {
        r.Use(mcpBearerAuthMiddleware(s.mcpServer.APIToken()))
        r.Handle("/*", s.mcpServer.SSEHandler())
    })
}
```

The MCP SSE server from `mcp-go` library implements `http.Handler`. When creating it with `server.NewSSEServer()`, set `BaseURL` to include the `/mcp` prefix so SSE endpoint URLs are correct.

**Important:** The `mcp-go` SSEServer's `NewSSEServer()` takes a `baseURL` parameter (or similar). Check the library API — the SSE handler registers paths like `/sse` and `/message`. When mounted under `/mcp`, these become `/mcp/sse` and `/mcp/message`. The `baseURL` must reflect this so clients receive correct endpoint URLs.

### Step 3: Update cmd/serve.go initialization

In `cmd/serve.go`, after the existing initialization (PostgreSQL, HNSW, etc.):

1. Check if `MCP_API_TOKEN` env var is set
2. If yes, create MCP server using existing shared resources:
   - `database.GetBookWriter()`, `database.GetTextVersionStore()`, `database.GetTextCheckStore()`, `database.GetEmbeddingReader()` — these are already registered by `serve` command
   - PhotoPrism client — create a dedicated session for MCP (separate from web session management)
3. Pass MCP server to `web.NewServer()` or call a setter
4. If `MCP_API_TOKEN` is not set, log info message: "MCP_API_TOKEN not set, MCP endpoint disabled" and skip

### Step 4: Auth middleware for MCP routes

Create a simple Bearer token middleware (move from `internal/mcp/server.go:authMiddleware` to reusable location):

```go
func mcpBearerAuthMiddleware(token string) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            auth := r.Header.Get("Authorization")
            if auth != "Bearer "+token {
                http.Error(w, "Unauthorized", http.StatusUnauthorized)
                return
            }
            next.ServeHTTP(w, r)
        })
    }
}
```

MCP routes use Bearer token auth (separate from session-based web auth). No session cookies involved.

### Step 5: Graceful shutdown

The existing `serve` command already handles graceful shutdown (signal handling, HNSW save, pool close). No additional shutdown logic needed for MCP — it's just routes on the same HTTP server.

### Step 6: Keep mcp-serve command as deprecated alias (optional)

Either:
- Remove `cmd/mcp_serve.go` entirely
- Or keep it but have it print a deprecation warning pointing to `serve` with `MCP_API_TOKEN`

Recommend: remove it to avoid confusion.

## Testing

1. Start `serve` WITHOUT `MCP_API_TOKEN` → MCP routes not available, web works normally
2. Start `serve` WITH `MCP_API_TOKEN=test123` → MCP available at `/mcp/sse`, web works normally
3. Test MCP auth: request without Bearer token → 401
4. Test MCP auth: request with correct Bearer token → MCP tools work
5. Test that existing web auth (session cookies) is unaffected
6. Test graceful shutdown saves HNSW indexes (unchanged behavior)
