## Cíl

Zkontrolovat, zda aktuální schéma vývojové PostgreSQL databáze (`photosorter` na `shared-postgres`) odpovídá tomu, co by vytvořily SQL migrace v `internal/database/postgres/migrations/` (001–020).

## Postup

1. **Zjisti stav migrací v DB** — tabulka `schema_migrations` (verze, dirty flag). Pokud neexistuje, znamená to, že migrace nebyly nikdy spuštěny přes golang-migrate.

2. **Projdi všechny migrace 001–020** a z nich odvoď očekávané schéma (tabulky, sloupce, typy, constrainty, indexy, extensions).

3. **Zjisti aktuální schéma z databáze:**
   - `\dt` — seznam tabulek
   - `\d+ <table>` — sloupce, typy, default, NOT NULL, constrainty pro každou tabulku
   - `\di` — indexy
   - `\dx` — extensions (pgvector, unaccent)
   - CHECK constraints, FOREIGN KEY constraints, UNIQUE constraints

4. **Porovnej a zapiš rozdíly:**
   - Chybějící tabulky nebo sloupce
   - Odlišné typy sloupců
   - Chybějící nebo přebývající indexy
   - Chybějící nebo odlišné constraints (CHECK, FK, UNIQUE, DEFAULT)
   - Chybějící extensions
   - Migrace, které nebyly aplikovány (dle `schema_migrations`)
   - Dirty migration flag

5. **Výstup:** Pokud jsou rozdíly, vypiš je přehledně a navrhni opravu (buď nová migrace, nebo manuální SQL). Pokud je vše v pořádku, potvrď konzistenci.

## Přístup k DB

```bash
docker exec shared-postgres psql -U postgres -d photosorter
```

Případně:
```bash
docker exec shared-postgres psql -U postgres -d photosorter -c "SQL"
```

## Důležité

- NEKOPÍRUJ migrace a NEMĚŇ schéma databáze — jen analyzuj a reportuj.
- Pokud zjistíš nesrovnalosti, navrhni opravu formou nové migrace SQL souboru, ale NEAPLIKUJ ji.
- Zkontroluj i `types.go` a `books.go` v `internal/database/`, zda Go kód neočekává sloupce/tabulky, které v DB chybí nebo naopak.
